# 架构说明

## 核心闭环

```text
项目资料 / 文字 / 图片
          ↓
  AI 发散（严格 10 项）
          ↓
 React Flow 气泡画布
          ↓ 用户显式收集
      创意总结草稿
          ↓ 编辑 / 保存版本
      完整项目计划
          ↓ 编辑 / 保存版本
统一 ExportDocument → DOCX / PDF / TXT / Markdown / JSON
```

AI 结果和用户选择分开存储：气泡的 `collected` 是人的判断；总结与计划保留 `sourceNodeIds`，便于回溯，但导出层会移除这些内部标识。

## 分层

### 界面层

`src/components/idea-bubble-app.tsx` 负责阶段编排。画布、收集栏、总结编辑器、计划编辑器、媒体入口和导出面板各自独立。桌面端使用三栏布局，移动端把左右栏变成 Sheet。

### 状态与持久化

Zustand store 是浏览器内的工作流状态源。用户更改进入带时间合并的自动保存；项目仓库通过 `idb` 写入 IndexedDB。撤销/重做只维护当前会话中的项目快照，不会把 UI 临时状态写入领域对象。

持久化边界是 `Project`：项目资料、原始输入、素材 data URL、节点/边、当前草稿、版本、导出预设和请求日志都可保存。当前阶段、面板开关和 loading 状态不持久化。

### AI 服务

客户端只提交结构化 JSON 到同源 Route Handler。服务端按任务选择模型，Vercel AI SDK 生成 Zod 约束的对象。发散结果去重后不足 10 项时只进行一次修复；仍不满足则返回可重试错误。

公共 POST 包装器统一处理：

- Content-Type 和请求体大小；
- Zod 输入校验；
- 按客户端与路由的内存限流；
- 基于请求体哈希的短期幂等；
- 超时、取消和不泄露内部异常的错误信封；
- `no-store` 与 `nosniff` 响应头。

这些内存结构只适合单实例 MVP。多实例部署应替换为 Redis/KV 或网关限流。

### 导出服务

客户端把项目快照、Preset 和同一批次时间发给导出路由。`buildExportDocument` 负责：

1. 清洗用户文本与文件名；
2. 按勾选顺序构建非空章节；
3. 解析当前草稿/最终版本优先级；
4. 删除内部 ID、AI 日志和 Provider 信息；
5. 统一封面、元数据、版式与素材描述。

实时预览、DOCX、PDF、TXT、Markdown 和 JSON 都消费这一个模型，因此章节选择和内容不会各自漂移。

## 关键约束

- 单项目最多 200 个节点，避免画布与 IndexedDB 无边界增长。
- 静态图片 3 MB、GIF 8 MB、视频 25 MB；当前持久化为 data URL，适合演示而非大规模素材库。静态图片的较低上限用于容纳 base64 膨胀，并确保视觉分析请求低于线上 Function 的 payload 限制。
- 图像分析请求使用更小的服务端体积上限；视频/GIF 分析明确不支持。
- API Key 只存在于服务端进程环境中。
- PDF/DOCX 使用随项目分发的 OFL 字体，避免依赖部署机器系统字体。

## 建议的生产化演进

1. 用 Postgres 保存项目元数据，Blob/S3 保存素材与导出文件。
2. 加入认证、项目级授权、CSRF/Origin 策略与审计日志。
3. 用队列处理大型导出和视频抽帧，返回短期签名下载链接。
4. 为 IndexedDB 项目添加 schema 版本和显式迁移。
5. 把限流与幂等迁移到共享存储，并增加请求成本与模型用量记录。
